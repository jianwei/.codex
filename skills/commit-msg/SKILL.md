# 智能提交信息生成 Skill

这个 skill 帮助开发者基于代码变更自动生成规范的 Git 提交信息，包含变更目的、影响范围和测试说明。

## 使用场景

当用户需要提交代码时，这个 skill 会：
1. 分析 git diff 中的代码变更
2. 理解变更的业务含义和技术影响
3. 生成结构化的提交信息
4. 引导用户确认并执行提交

## 执行流程

### 第一步：检查 Git 状态

```bash
git status
```

检查项目：
- 是否有未提交的变更
- 哪些文件被修改/新增/删除
- 是否有已暂存的文件

### 第二步：确定要提交的文件

**重要：** 先询问用户要提交哪些文件

#### 2.1 检查已暂存的文件
```bash
git diff --cached --name-status
```

#### 2.2 检查未暂存的修改
```bash
git diff --name-status
```

#### 2.3 交互式选择

如果检测到多个修改的文件，询问用户：

```
检测到以下文件有修改：

已暂存（将被提交）：
  M  path/to/file1.java
  M  path/to/file2.java

未暂存（不会被提交）：
  M  path/to/file3.java
  M  path/to/file4.java

请确认：
1. 只提交已暂存的文件（推荐）
2. 选择特定文件暂存并提交
3. 暂存所有修改并提交

请选择（1/2/3）：
```

**用户选择后的处理：**

- **选择1**：继续使用 `git diff --cached`
- **选择2**：询问要暂存哪些文件，然后执行 `git add`
- **选择3**：执行 `git add -A`（需谨慎，确认没有敏感文件）

### 第三步：获取代码差异

根据用户选择，执行相应的 diff：

**如果有已暂存的文件：**
```bash
git diff --cached
```

**如果选择了特定文件：**
```bash
git diff [选择的文件列表]
```

### 第四步：分析代码变更

仔细分析以下维度：

#### 1. 变更类型识别
- **feat**: 新功能
- **fix**: Bug 修复
- **refactor**: 重构（不改变功能）
- **perf**: 性能优化
- **style**: 代码格式调整
- **test**: 测试相关
- **docs**: 文档更新
- **chore**: 构建/工具变更

#### 2. 变更目的和背景分析
通过代码变更回答：
- **为什么做这个变更？**
  - 解决什么问题？
  - 优化什么流程？
  - 添加什么能力？
- **业务背景是什么？**
  - 是否关联需求单（JIRA/AMID）
  - 用户痛点是什么
- **技术背景是什么？**
  - 代码设计问题
  - 性能瓶颈
  - 技术债务

#### 3. 影响范围评估
识别：
- **文件级影响**：修改了哪些文件
- **模块级影响**：影响哪些功能模块
- **类/方法级影响**：核心变更在哪些类/方法
- **依赖影响**：是否影响其他模块调用
- **破坏性变更**：
  - 是否改变了 API 签名
  - 是否删除了公共方法
  - 是否修改了数据结构

#### 4. 测试说明生成
根据变更类型推荐：
- **单元测试**：
  - 是否需要新增测试
  - 是否需要更新现有测试
  - 覆盖率要求
- **集成测试**：
  - 接口调用测试
  - 数据库交互测试
- **手工测试步骤**：
  - 功能验证步骤
  - 边界条件验证
  - 异常场景验证

### 第五步：生成提交信息

按以下格式生成：

```
<类型>: <简短描述>（不超过50字符）

## 变更目的和背景
[基于代码分析，清晰说明为什么做这个变更]
- 问题描述：...
- 解决方案：...

## 变更影响范围
- 影响文件：
  - path/to/file1.java
  - path/to/file2.java
- 影响模块：[模块名称]
- 核心变更：[核心类/方法]
- 破坏性变更：有/无
  [如果有，详细说明]

## 测试说明
- 单元测试：已添加/需补充/不需要
  - [具体测试用例说明]
- 手工测试步骤：
  1. [步骤1]
  2. [步骤2]
  3. [验证预期结果]

## 相关链接
JIRA: [如果能从代码注释或分支名推断出，填写对应的需求单号]

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
```

### 第六步：用户确认

展示生成的提交信息，询问用户：
1. 是否需要修改
2. 是否要补充信息
3. 是否确认提交

### 第七步：执行提交

用户确认后，根据情况：

**如果文件已暂存：**
```bash
git commit -m "$(cat <<'EOF'
[生成的提交信息]
EOF
)"
```

**如果文件未暂存，询问是否要全部暂存：**
```bash
git add [用户选择的文件]
git commit -m "$(cat <<'EOF'
[生成的提交信息]
EOF
)"
```

## 特殊场景处理

### 场景1：多个独立变更
如果 diff 显示多个不相关的变更，建议：
```
检测到多个独立的变更：
1. [变更1描述]
2. [变更2描述]

建议：分别提交以保持提交历史清晰
是否要分别处理？
```

### 场景2：没有变更
```
当前没有待提交的变更。
请先修改代码，然后再使用此 skill。
```

### 场景3：变更太大
如果变更超过 500 行，建议：
```
检测到大量变更（[具体行数]行）。
建议：
1. 拆分成多个小的提交
2. 或者先审查变更是否合理

是否继续生成提交信息？
```

## 最佳实践

### 提交信息质量标准
- **简洁性**：标题不超过 50 字符
- **清晰性**：目的明确，不含糊
- **完整性**：包含必要的三个部分
- **可追溯**：关联需求单或问题

### 代码变更建议
- **原子性**：一次提交只做一件事
- **完整性**：变更要完整，不遗漏文件
- **可测试**：变更要可验证

```

- 问题：推荐景点查询逻辑在两处重复，违反 DRY 原则

  - 新增 buildScenicInfoRequest() 方法（构造查询请求）

  1. 测试无门票景点的兜底逻辑
```

## 注意事项

1. **保护敏感信息**：提交信息中不要包含密码、密钥等敏感信息
2. **语言选择**：根据团队习惯选择中文或英文
3. **关联需求单**：尽量关联 JIRA/需求单号
4. **代码审查**：大的变更建议先创建 PR
